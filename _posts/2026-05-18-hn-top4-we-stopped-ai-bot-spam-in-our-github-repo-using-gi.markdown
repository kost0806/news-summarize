### 개요

Archestra.ai는 자사 GitHub 저장소에서 AI 봇 스팸의 급격한 증가를 경험했다. 현상금(bounty) 이슈 하나에 253개의 봇 댓글이 달리고, 단일 기능에 대해 27개의 미검증 풀 리퀘스트(PR)가 제출되는 상황에 이르렀다. 여러 해결책을 시도한 끝에, GitHub이 "이전 기여자(prior contributor)"를 정의하는 방식을 활용하는 영리한 해결책을 발견했다.

### 주요 내용

#### AI 봇 스팸의 실상

오픈소스 저장소에서 AI를 활용한 자동화 스팸이 새로운 문제로 부상하고 있다. Archestra.ai가 경험한 구체적인 피해:

- 현상금(bounty) 이슈 단 하나에 **253개의 봇 댓글** 폭주
- 단일 기능 구현을 위한 **27개의 풀 리퀘스트** 제출 — 대부분 실제 테스트 없음
- 저품질 AI 생성 코드로 인한 리뷰 부담 급증
- 실제 기여자들의 기여가 봇 스팸에 묻히는 현상

#### 시도했다가 실패한 해결책들

**1단계: London-Cat 봇 (평판 점수 시스템)**
GitHub 계정의 평판 점수(reputation score)를 기반으로 스팸을 필터링하는 봇을 도입했다. 문제점: 평판 점수 측정 자체가 부정확했고, 새로운 정상 기여자들도 차단되는 오탐(false positive)이 발생했다.

**2단계: AI 보안관 봇 (AI Sheriff)**
AI가 봇 여부를 판단하여 PR을 자동으로 닫는 "AI 보안관" 시스템을 구현했다. 문제점: 정상적인 PR을 잘못 닫는 사례가 발생하여 실제 기여자들의 반발을 샀다.

#### 최종 해결책: Git --author 플래그 활용

핵심 발견: **GitHub은 "이전 기여자"를 메인 브랜치(main branch)에 커밋(commit)을 authored한 계정으로 정의한다.**

이를 활용한 해결책:

```bash
git commit --author="username <id+username@users.noreply.github.com>"
```

승인된 기여자들을 온보딩(onboarding) 과정을 거친 후, 위 명령어로 메인 브랜치에 해당 사용자를 author로 하는 커밋을 추가한다. 이렇게 하면:

1. 해당 사용자는 실제 코드를 커밋하지 않아도 GitHub의 "이전 기여자"가 됨
2. GitHub의 자동 승인 규칙에 따라 이전 기여자만 댓글/PR 제출 가능하게 설정
3. 화이트리스트(whitelist)에 없는 계정의 PR과 댓글은 자동으로 보류(hold)

이 방식의 장점:
- **추가 인프라 불필요**: GitHub의 기존 기능만 활용
- **오탐 최소화**: 온보딩을 통해 검증된 사용자만 허용
- **확장 가능**: 기여자가 늘어날수록 커뮤니티가 자체적으로 성장

### HN 커뮤니티 반응

HN 점수: 374점 | 댓글: 180개

이 게시물은 오픈소스 유지관리자들 사이에서 큰 공감을 불러일으켰다:

**광범위한 공감**: 많은 저장소 관리자들이 비슷한 AI 봇 스팸 문제를 경험하고 있다는 댓글이 쏟아졌다. 특히 현상금이 걸린 이슈나 인기 있는 프로젝트에서 AI 자동화 스팸이 급격히 증가했다는 증언이 이어졌다.

**GitHub 기능 활용의 영리함**: `--author` 플래그를 통한 화이트리스트 관리가 GitHub의 기존 메커니즘을 창의적으로 활용했다는 점에서 높은 평가를 받았다. "해킹(hack)이지만 영리한 해킹"이라는 반응이 많았다.

**GitHub의 책임 논쟁**: 이런 문제를 해결하는 것이 개별 저장소 관리자의 책임인지, GitHub 플랫폼이 기본 제공해야 하는 기능인지에 대한 토론이 있었다. GitHub이 AI 봇 스팸에 대한 더 강력한 기본 도구를 제공해야 한다는 주장이 많았다.

**온보딩의 부담**: 모든 기여자가 온보딩 과정을 거쳐야 한다는 점이 일부 소규모 프로젝트에는 과도한 장벽이 될 수 있다는 우려도 제기되었다.

### 시사점

AI 코딩 도구의 보급으로 오픈소스 생태계에 새로운 도전이 생겨났다. AI가 자동으로 생성한 저품질 기여(contribution)들이 저장소 관리자의 부담을 가중시키고, 진정성 있는 기여자들의 목소리를 묻히게 만든다. 이 사례는 플랫폼이 제공하는 기존 메커니즘을 창의적으로 조합하여 새로운 문제를 해결할 수 있음을 보여준다. 오픈소스 프로젝트 관리자들은 기여 품질과 커뮤니티 건전성을 유지하기 위한 새로운 게이트키핑(gatekeeping) 전략을 적극적으로 도입할 필요가 있다.
