### 개요

이 글은 현실에서 C 코드의 이식성(portability)이 얼마나 복잡한 문제인지를 구체적인 사례로 분석한다. ISO C 표준을 완벽히 준수하는 코드를 작성하는 것은 이론적으로는 가능하지만 **현실의 거의 모든 C 코드는 비표준 동작과 언어 확장에 의존한다**는 점을 지적한다.

### 주요 내용

**표준 C의 한계**

ISO C 표준은 여러 중요한 영역에서 구현 정의(implementation-defined) 또는 미정의(undefined) 동작을 허용한다. 이로 인해 실제 코드는 다음과 같은 컴파일러별 확장에 의존하게 된다:

- **GCC 스타일 인라인 어셈블리**: `__asm__` 구문은 GCC와 Clang에서 지원되지만 다른 컴파일러에서는 지원하지 않는 경우가 많다
- **`__attribute__` 확장**: 함수 속성, 섹션 지정, 정렬 요구사항 등을 지정하는 GCC 확장
- **`__builtin_*` 함수들**: 컴파일러 내장 함수들은 표준 라이브러리에 없지만 성능 최적화에 필수적인 경우가 많다
- **가변 길이 배열(VLA)**: C99에서 선택적(optional)으로 변경된 이후 지원 여부가 불분명해짐

**실제 사례: SDL_endian.h**

저자는 SDL의 바이트 순서 감지 헤더를 예시로 든다. 이 헤더의 기능 감지 로직은 GCC 스타일 인라인 어셈블리를 지원하지 않는 컴파일러도 GCC나 Clang이라고 가정하는 문제가 있다. MSVC나 icc 같은 대안 컴파일러를 사용할 때 예상치 못한 코드 경로로 빠지는 이유가 여기에 있다.

**대안 컴파일러의 도전**

- **TCC (Tiny C Compiler)**: GCC 확장의 일부만 지원
- **MSVC**: Windows 환경에서 표준이지만 GCC 확장과 호환성이 떨어짐
- **Chibicc, cproc** 등 실험적 컴파일러: 표준 준수는 높지만 실제 코드베이스와의 호환성이 낮음

### 시사점

C 언어의 이식성 신화와 현실의 간극을 명확히 보여주는 글이다. 임베디드 시스템, 크로스 컴파일, 레거시 코드 유지보수를 다루는 개발자라면 반드시 숙지해야 할 내용을 담고 있다.
